# Hermes Chat-first 问道页与 shadcn/i18n 整改设计

## 背景

Ice Cola 的 `packages/client` 问道页不是普通聊天窗口，而是连接 Hermes 能力的工作台。它需要承载 WebSocket Gateway、`hermes.send` 流式响应、模型与灵气计费、专家、MCP、技能、插件、多模态附件和工具执行状态。同时，`packages/client` 与 `packages/admin` 仍存在部分原生控件、硬编码文案和组件规范不一致问题，需要按高频路径优先整改。

本设计采用用户已确认的方向：**Chat-first 聚焦体验**。对话区保持最大视觉优先级，Hermes 能力通过轻量能力栏、弹层和消息内执行状态呈现。

## 目标

1. 将问道页定位为 Hermes Chat-first 工作台，而不是泛聊天 UI。
2. 最大化对话区域，减少常驻配置面板对纵向空间的占用。
3. 输入框默认 1 行，最多自适应到 3 行，发送成功后恢复 1 行高度。
4. 支持专家、MCP、技能、插件、多模态附件、模型和灵气预估。
5. 优先用本地 shadcn/ui 组件替换高频路径里的原生控件。
6. 清理 client/admin 高频页面硬编码中英文，补齐 locale key。
7. 保持单文件不超过项目 500 行限制，拆分 `Chat.tsx` 职责。
8. 完成后用单测、build 和 Playwright MCP 多屏 E2E 报告验证。

## 非目标

1. 不重写 Hermes/Gateway 协议。
2. 不一次性整改全仓所有原生标签。
3. 不改变现有认证、团队、灵气扣费和退款语义。
4. 不引入新的 UI 组件库；只使用项目本地 shadcn/ui 风格组件。
5. 不把问道页改成 Command Center 常驻大面板布局。

## 推荐方案

采用 **A1：Chat-first + Hermes 能力栏**。

- 页面主体仍以会话消息为核心。
- 左侧会话列表在宽屏常驻，窄屏抽屉化。
- 底部输入区上方放一条轻量 Hermes 能力栏。
- 能力栏承载 Gateway/Hermes 状态、模型、灵气、专家、MCP、技能、插件、多模态入口。
- 详细选择流程通过 Popover、Sheet、Dialog 或 Drawer 展开，不常驻挤占对话区。
- 工具调用、MCP 调用、技能执行和错误状态在消息内以 Badge、Card、Collapsible 等方式呈现。

该方案比右侧能力抽屉更符合“尽可能多的对话部分”，也比 Command Center 更轻量。

## Client 问道页信息架构

### 桌面宽屏

- 左侧：会话列表和新建会话入口。
- 中央：消息流、空状态、流式响应和工具执行块。
- 底部：Hermes 能力栏 + 输入框 + 发送区。
- 顶部：只保留必要标题、连接状态和当前团队上下文。

### 中等屏幕

- 会话列表收窄，仅保留核心会话信息。
- 能力栏保持一行，但允许低优先级入口折叠到“更多”。
- 模型、灵气、专家保持可见；MCP、技能、插件、多模态可通过分组入口展开。

### 平板与窄屏

- 会话列表进入 Sheet。
- Hermes 状态压缩到顶部短状态条。
- 能力栏变成底部能力托盘，点击后以 Sheet 展示分组能力。
- 输入框和发送按钮始终固定在底部可达区域。

### 低高度窗口

- 压缩 header、能力栏和输入区间距。
- 优先保证消息区高度。
- 能力详情默认折叠，避免挤压对话。

## 组件拆分

### `ChatWorkspace`

负责问道页整体布局：侧栏、消息区、底部 composer、响应式断点状态。它只编排子组件，不直接处理具体选择器细节。

### `HermesCapabilityBar`

展示 Hermes 能力状态和入口：

- Gateway/Hermes 连接状态。
- 当前模型。
- 灵气余额与发送前预估。
- 专家选择。
- MCP 服务器选择。
- 技能入口。
- 插件入口。
- 多模态附件入口。

它应使用本地 shadcn/ui 的 Button、Badge、Popover、Sheet、Separator、Tooltip 等组件。宽屏显示多个短按钮，窄屏折叠为分组入口。

### `ChatComposer`

负责输入体验：

- 使用本地 shadcn/ui Textarea 或项目等价组件。
- 默认 1 行高度。
- 输入内容增长时最多 3 行。
- 发送成功后清空内容并恢复 1 行高度。
- Enter 发送，Shift+Enter 换行。
- 禁用状态需要区分 Gateway 未连接、正在发送、团队缺失。
- 多模态附件入口与输入框同一操作区。

### `HermesMessageList`

负责消息列表、空状态、滚动到底部和流式消息展示。它不持有 Gateway 协议细节，只接收标准化消息和执行事件。

### `HermesExecutionBlock`

展示 Hermes 执行过程：

- 工具调用中。
- 工具调用成功。
- 工具调用失败。
- MCP 调用信息。
- 技能/插件执行状态。
- 可折叠查看细节。

该组件替代消息项中散落的硬编码工具文案和原生按钮。

### `ChatEmptyState`

空状态不再表达“开始聊天”，而是表达“连接 Hermes 能力”：

- 可提示选择模型、专家或 MCP。
- 可展示 Gateway/Hermes 就绪状态。
- 可保留少量快捷提示，但不占用过多空间。

## shadcn/ui 整改范围

### Client 优先范围

1. 问道页输入区、能力栏、消息操作按钮。
2. `ChatMessageItem` 中编辑、复制、重发、工具调用折叠等动作。
3. 专家、MCP、技能、插件入口中的高频表单控件。
4. 多模态附件上传和删除动作。

### Admin 优先范围

1. 兑换码列表、详情、创建和状态操作。
2. 灵气账本筛选、表格、分页和详情展示。
3. 管理后台布局中高频按钮、输入、选择器和表格。

### 替换原则

- 原生 `button` 替换为本地 `Button`。
- 原生 `input` 替换为本地 `Input`。
- 原生 `textarea` 替换为本地 `Textarea`。
- 原生 `select` 替换为本地 `Select`。
- 原生 `table` 替换为本地 `Table`。
- 确认操作必须使用项目 Dialog / ConfirmDialog，不使用 `window.confirm()`。
- 仅在语义 HTML 更合适且无需组件样式时保留原生元素。

## 国际化整改范围

### Client locale

在 `chat` 命名空间补齐：

- Hermes/Gateway 连接状态。
- 能力栏入口。
- 模型与灵气预估。
- 专家、MCP、技能、插件、多模态。
- 工具调用状态、成功、失败和未知错误。
- 输入禁用原因。
- 空状态和响应式折叠入口。

### Admin locale

补齐或整理：

- 兑换码列表、详情、创建、复制、状态变更文案。
- 灵气账本筛选、表格列、空状态、错误状态。
- 通用表单动作、确认弹窗、加载和失败提示。

### 规则

- 高频 UI 不保留硬编码中文或英文。
- locale key 按页面和功能分组。
- 删除明显重复 key，避免同名覆盖。
- 不做全仓低频文案大扫除，防止范围失控。

## Hermes 数据与交互流

1. 用户进入问道页。
2. 页面读取团队、Gateway 连接、Lingqi 状态和可用模型。
3. 能力栏展示 Hermes/Gateway 就绪状态。
4. 用户选择模型、专家、MCP、技能、插件和附件。
5. 用户输入消息，发送前展示灵气预估。
6. 点击发送或 Enter 触发 `hermes.send`。
7. Gateway 返回流式 delta、final、error 和工具事件。
8. 消息列表实时展示响应内容和执行块。
9. 成功时 composer 清空并恢复 1 行。
10. 失败时保留必要上下文，展示可理解错误和可重试动作。

## 错误处理

- Gateway 未连接：输入区禁用，显示连接状态和重连入口。
- 团队缺失：提示创建或加入团队，不允许发送。
- 灵气不足：发送前阻断，展示余额和预估差额。
- Hermes 请求失败：消息内显示失败块，可重试。
- 工具/MCP/技能失败：在对应执行块显示失败原因，不吞掉错误。
- 附件不支持：在选择时立即提示格式或大小限制。
- 流式中断：保留已收到内容，标记为中断状态。

错误文案必须走 i18n，内部错误细节不直接暴露给终端用户。

## 可访问性

- 所有能力入口需要可聚焦和键盘触达。
- Popover、Sheet、Dialog 使用可访问组件实现焦点管理。
- 图标按钮必须有 aria-label 或可见文本。
- 工具执行状态需要文本说明，不能只靠颜色。
- 输入框 Enter/Shift+Enter 行为保持一致且可预期。

## 测试与验证

### 单元/组件测试

- `ChatComposer`：1 行默认、最多 3 行、发送后复位、Enter/Shift+Enter 行为。
- `HermesCapabilityBar`：不同屏幕状态下关键入口存在或折叠。
- `HermesExecutionBlock`：执行中、成功、失败、未知工具名展示。
- i18n：关键 locale key 在 zh/en 中均存在。

### 构建与静态检查

- 跑 client/admin 相关测试。
- 跑 client/admin build。
- 修复 TypeScript 错误，不新增 `any`。

### Playwright MCP E2E

生成 `reports/问道页_Hermes_Chat_First_E2E_TEST_REPORT_YYYYMMDD.md`，覆盖：

- 1440、1024、768、375 宽度布局。
- 低高度窗口布局。
- 输入框 1 行、3 行上限、发送后复位。
- Hermes/Gateway 未连接状态。
- 模型、灵气、专家、MCP、技能、插件、多模态入口可见或可展开。
- 工具执行块成功和失败状态。

## 实施边界

实施应分为小步提交候选，但不要求在设计阶段提交：

1. 盘点现有 client/admin 原生控件和硬编码文案。
2. 补齐缺失的本地 shadcn/ui 基础组件。
3. 拆分问道页组件并实现 Chat-first 布局。
4. 接入 Hermes 能力栏和 composer 行为。
5. 整改消息项工具执行展示和 i18n。
6. 整改 admin 高频兑换码/灵气页面组件和 i18n。
7. 补测试、跑 build、执行 Playwright MCP 并生成报告。

## 风险与约束

- `Chat.tsx` 当前职责较多，必须拆分以满足 500 行文件限制。
- 根目录缺少标准 `components.json`，shadcn 组件同步需要按 workspace 现状谨慎处理。
- Client 的 `test` 脚本可能是 watch 模式，验证阶段需要使用非 watch 命令。
- Hermes、Gateway、Lingqi 和多模态都涉及用户输入或外部服务，后续实现需要安全审查。
- 多屏布局容易产生底部输入区遮挡，E2E 必须覆盖低高度和窄屏。

## 验收标准

1. 问道页视觉和交互符合 Chat-first Hermes 工作台定位。
2. 对话区域比当前布局获得更高优先级，常驻配置面板被能力栏/弹层替代。
3. 输入框默认 1 行、最多 3 行、发送后恢复 1 行。
4. 专家、MCP、技能、插件、多模态、模型和灵气入口都可访问。
5. client/admin 高频路径原生控件被本地 shadcn/ui 组件替换或有明确保留理由。
6. 高频 UI 文案进入 zh/en locale，无明显新增硬编码文案。
7. 相关源文件保持 500 行以内。
8. 单测、build 和 Playwright MCP E2E 报告完成，未遗留 CRITICAL 或 HIGH 问题。
